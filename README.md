# MDSP Project — Matrix/Vector Multiplication on PYNQ (AXI-Stream + DMA)

This project implements streamed matrix–vector multiplication on FPGA and evaluates it from Python on a PYNQ platform using AXI DMA. Inputs are generated in Python/NumPy, streamed to the hardware kernel, and outputs are streamed back and compared against a NumPy reference.

The design supports **floating-point** and **fixed-point** data paths (quantized to integer “raw codes” for streaming).

## Project overview

Given:
- A matrix **A** of shape `(N, N)`
- A set of input vectors **X** of shape `(NS, N)`

Compute for each sample:
- **Y = X @ A.T** (i.e., `y_i = A * x_i` depending on chosen convention)

On the FPGA:
- Matrix coefficients and input vectors are sent via **AXI-Stream**.
- The kernel produces output vectors via **AXI-Stream**.
- **AXI DMA** moves data between PS (ARM) memory and the stream ports.

On the PS (Python):
- NumPy computes a software reference output.
- DMA streams input data to the FPGA kernel.
- DMA receives output data back.
- Outputs are converted (if fixed-point) and compared to reference.

## Hardware datapath

The notebook assumes the overlay contains:

- `axi_dma_0` — AXI DMA (MM2S + S2MM)
- `matmul_stream_2_0` — matrix/vector multiply streaming kernel
- `axi_timer_0` — hardware timer for cycle counting

Dataflow:

PS memory → DMA (send) → AXI-Stream → **matmul_stream** → AXI-Stream → DMA (recv) → PS memory

The kernel is configured via AXI-Lite registers. In the notebook, the sample count is written via:

- `matmul_stream.write(0x10, NS)`  (sets number of samples)
- `matmul_stream.write(0x00, 0x01)` (starts IP)

> Register offsets may differ depending on your HLS export; adjust if necessary.

## Data formats (float vs fixed)

### 1) Floating-point mode

- DMA buffers use `np.float32`.
- Each 32-bit DMA word is a float sample.

In the notebook:
- `USE_FIXED = False`
- `dma_dtype = np.float32`
- no quantization is applied

### 2) Fixed-point mode (streaming raw codes)

In fixed-point mode, the DMA still streams **32-bit words**, but each word holds a **signed fixed-point integer code** (two’s complement). The FPGA interprets these codes as `ap_fixed<...>` values.

For `F` fractional bits:

- Quantization (float → code):
  - `q = round(x * 2^F)`
  - saturate `q` to the integer range supported by your fixed type
- Reconstruction (code → float):
  - `x ≈ q / 2^F`

In the notebook:
- `USE_FIXED = True`
- DMA buffers use `np.int32`
- conversion is handled by:
  - `format_input_data(...)`  (float → int codes)
  - `format_output_data(...)` (int codes → float)

**Important:** `FRACTIONAL_BITS` must match what the kernel uses internally.
Example: if hardware uses `ap_fixed<16,4>`, then `F = 16-4 = 12`.

## Test-vector generation

The notebook generates random data and normalizes it:

- `generate_matrix(n)`:
  - generates `A ~ N(0,1)`
  - normalizes by Frobenius norm

- `generate_vectors(n, ns)`:
  - generates `X ~ N(0,1)` for `ns` samples
  - normalizes each vector by its L2 norm

Reference output is computed as:

- `Y = X @ A.T`

The notebook can save:
- matrix to `test/matrices/{n}.txt`
- inputs to `test/input/{n}.txt`
- outputs to `test/output/{n}.txt`

## Running on PYNQ

1. Copy the project to the PYNQ board (or open it in JupyterLab on the board).
2. Ensure the expected overlay bitstream exists (default in the notebook):
   - `matmulfloat.bit` (or update the filename in the notebook)
3. Open `matmulgauta.ipynb` and run all cells.

Key parameters you can edit in the notebook:
- `N` — matrix dimension (e.g., 64)
- `NS` — number of input samples/vectors
- `USE_FIXED` — set `True` for fixed-point streaming
- `FRACTIONAL_BITS` — must match hardware fixed-point format

## Accuracy checking

After DMA receive:
- outputs are reshaped to `(NS, N)`
- error is computed vs NumPy reference:
  - `error = y_calculated - output_vectors`
  - `l2_errors = norm(error, axis=1)`
  - reports `Average L2 error`

Notes:
- In fixed-point mode, some error is expected due to quantization and limited precision.
- If error is unexpectedly large, the most common cause is **mismatched fixed-point scaling**:
  - Python `FRACTIONAL_BITS` does not match hardware
  - hardware interprets stream words differently (float bits vs fixed codes)

## Troubleshooting

### 1) Output is garbage in fixed mode
Likely causes:
- sending float32 but hardware expects fixed codes
- `FRACTIONAL_BITS` mismatch
- kernel expects narrower codes in LSBs (e.g., 16-bit) but Python sends full-range 32-bit saturation

Fix:
- confirm the kernel’s `ap_fixed<W,I>` and compute `F = W-I`
- update Python scaling and saturation to match `W` (not always 32)

### 2) DMA transfer hangs
Common issues:
- mismatch between expected number of words and what kernel produces
- kernel not started (missing `ap_start`)
- incorrect IP name / overlay mismatch

Fix:
- verify `N_INPUT_WORDS` and `N_OUTPUT_WORDS`
- ensure kernel is configured and started before transfers

### 3) Large error even in float mode
Check:
- matrix/vector ordering (row-major vs column-major)
- whether hardware computes `A*x` or `x*A` (transpose convention)

## Notes / assumptions

- DMA words are 32-bit wide.
- In fixed mode, the stream carries signed integer codes (two’s complement).
- The notebook currently loads `matmulfloat.bit`; switch to a fixed overlay when you have it.
- The kernel register map is assumed to use offset `0x10` for `NS`.
